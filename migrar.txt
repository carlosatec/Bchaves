# PLANO DE MIGRAÇÃO - Qchaves v2.0
## OBJETIVO: Bitcoin Only - Máxima Performance CPU

Este documento define a reescrita do zero do Qchaves com foco absoluto em performance máxima para CPU, otimizações de hardware, e apenas Bitcoin (sem ETH).

## 1. FILOSOFIA DO PROJETO

### Puzzles de Teste
Pasta `Puzzles/` contém arquivos de teste com puzzles reais:

### 1.1 Princípios de Performance
- **CPU-first**: Toda otimização visa throughput de CPU
- **Memory-first**: Minimizar memory bandwidth, maximizar cache
- **SIMD-friendly**: Preparado para AVX2/AVX512
- **Branch-free**: Minimizar branch misprediction
- **No allocations in hot path**: Pre-allocation, pools de memória

### 1.2 Stack Tecnológico
- C++17 para arquitetura e infraestrutura
- C acelerado para loops críticos (inline assembly onde necessário)
- libsecp256k1 com GLV endomorphism (sempre ativo)
- Sem dependências desnecessárias (não precisa de GMP)

## 2. PIPELINE DE CRIPTOGRAFIA (HOT PATH)

Este é o código mais executado - deve ser极致 (extremamente otimizado):

```
private_key (32 bytes)
    │
    ▼ [Scalar Multiplication - GLV ativo]
public_key uncompressed (64 bytes) ──► public_key compressed (33 bytes)
    │                                        │
    ▼ [SHA256 + RIPEMD160]                 ▼ [SHA256]
hash160 uncompressed (20 bytes)       hash160 compressed (20 bytes)
    │                                        │
    ▼ [Base58Check]                       ▼ [Base58Check]
address uncompress (34-35 chars)      address compress (26-34 chars)
    │
    ▼ [memcmp against targets]
FOUND / NOT FOUND
```

### 2.1 Otimizações Obrigatórias

#### Scalar Multiplication (secp256k1)
- **GLV Endomorphism**: Sempre ATIVO (não opcional)
- **wNAF (width-4 Non-Adjacent Form)**: Reduzir pontos
- **Jacobian coordinates**: Evitar inversões modulares
- **Precomputed tables**: Janelas de 4, 8, ou 16 bits
- **Batch point addition**: Múltiplas somas de uma vez

#### Hash Operations
- **Inline functions**: Sem overhead de chamada
- **CPU intrinsics**: SHA-NI se disponível (SHA extensions)
- **Unrolled loops**: Sem branches no hash
- **Lookup tables**: Pequenos, cacheáveis

#### Address Generation
- **Base58 lookup table**: 58-entry, cache-friendly
- **memcmp**: Comparação SIMD-friendly
- **Avoid string operations**: Trabalhar com bytes até output final

### 2.2 Código de Exemplo - Hot Path
```c
// core/scalar_mul.h -_inline para máximoinline
static inline void scalar_mul_glv(const uint8_t* scalar, uint8_t* result) {
    // GLV decomposition: k = k1 + k2*λ
    // Two half-sized multiplications
    // ~50% faster que multiplication normal
    
    __int128 k1, k2;
    glv_decompose(scalar, &k1, &k2);  // ~5% do tempo
    
    point p1 = base_mul(k1);           // ~45% do tempo
    point p2 = base_mul(k2);           // ~45% do tempo
    
    // p1 + φ(p2) = P
    point_add_endo(p1, p2, result);    // ~5% do tempo
}
```

## 3. ARQUITETURA DE MEMÓRIA

### 3.1 Memory Pool System
```cpp
// pool.hpp - Alocador pré-reservado para evitar malloc em runtime
class KeyPool {
    alignas(64) uint8_t buffer[64 * 1024 * 1024];  // 64MB aligned
    size_t offset = 0;
    
    template<typename T>
    T* alloc(size_t count = 1) {
        T* ptr = (T*)(buffer + offset);
        offset += sizeof(T) * count;
        return ptr;
    }
};
```

### 3.2 Evitar Cache Misses
- **Prefetching**: __builtin_prefetch() em loops
- **Data layout**: Estruturas de arrays (SoA) para SIMD
- **Thread-local**: Cada thread tem sua própria memória
- **Lock-free**: Sem mutex no hot path

### 3.3 Pre-hash de Alvos

## 4. THREADING (MÁXIMA EFICIÊNCIA)

### 4.1 Thread Strategy
- **One thread per physical core**: Hiperthreading pode ajudar marginally
- **Work stealing**: Threads ociosas pegam trabalho de outras
- **Thread-local state**: Sem compartilhamento de estado mutável
- **Atomic counters apenas**: Para métricas, não para hot path

### 4.2NUMA Awareness
```cpp
// Bind thread to NUMA node
void pin_thread(int core_id) {
    cpu_set_t cpuset;
    CPU_ZERO(&cpuset);
    CPU_SET(core_id, &cpuset);
    pthread_setaffinity_np(pthread_self(), sizeof(cpu_set_t), &cpuset);
}
```

### 4.3 Batch Processing
```cpp
// Generate N keys, compute N pubkeys, check N addresses
// Minimiza overhead de thread coordination
const size_t BATCH_SIZE = 1024;  // Tunar por CPU

void process_batch(ThreadState* state) {
    // Gerar batch
    for (int i = 0; i < BATCH_SIZE; i++)
        state->keys[i] = next_key(&state->rng);
    
    // Compute todos pubkeys
    for (int i = 0; i < BATCH_SIZE; i++)
        scalar_mul_glv(state->keys[i], state->pubkeys[i]);
    
    // Hash todos
    for (int i = 0; i < BATCH_SIZE; i++)
        hash160(state->pubkeys[i], state->hashes[i]);
    
    // Comparar todos
    for (int i = 0; i < BATCH_SIZE; i++)
        if (check_hit(state->hashes[i])) found();
}
```

## 5. MODOS DE BUSCA

### 5.1 Address Search (Ranges pequenos até 2^40)
```
Sequential:  start → end (incremental)
Backward:   end → start (decremental)
Both:       Metade das threads em cada direção
Random:     Chaves aleatórias no range
```

### 5.2 BSGS (Ranges 2^32 - 2^64)
**Modo suportado:**
- **Sequential** (padrão único): Baby-step -> Giant-step sequencial

**Opcional (enhanced):**
- **-l compress|uncompress|both**: Buscar apenas compress, uncompress, ou ambos (semelhante ao Address mode)

**Nota técnica:** BSGS usa tabela pré-computada, então backward/both/random não se aplicam. Diferente do Address mode onde tem sentido (geração de chaves sequencial é possível).

**Características técnicas:**
- Tabela pré-computada (hash table linear)
- GLV endomorphism ativo
- **Switch automático**: Hash table para <1M alvos, Cuckoo filter para >1M alvos
- MMAP para tabela grande (se RAM permitir)

**Presets de tabela BSGS:**
| Preset | RAM | -k | Uso Recomendado |
|--------|-----|-----|-----------------|
| tiny | 2GB | 256 | Laptop/baixa RAM |
| small | 4GB | 512 | Desktop padrão |
| medium | 8GB | 1024 | Workstation |
| large | 16GB | 2048 | Servidor |
| huge | 32GB+ | 4096+ | High-end |

### Filtros de Busca - Análise

#### Address Search (Buscar endereços BTC)
Para o modo Address, o número de alvos é geralmente **pequeno** (puzzles têm tipicamente 1-100 endereços). Porém, para aplicações com milhões de endereços:

| Alvos | Método Recomendado |
|-------|-------------------|
| < 1.000 | Array linear (memcmp) - mais simples e rápido |
| 1.000 - 100K | Hash table (lookup O(1)) |
| 100K - 1M | **Cuckoo filter** (melhor que Bloom para este rango) |
| > 1M | Bloom filter + verificação extra |

**Address: Cuckoo > Bloom** porque:
- Zero false positives (Cuckoo é exato, Bloom não)
- Lookup mais rápido
- Menos verificação extra necessária

#### BSGS (Buscar public keys)
| Alvos | Método Recomendado |
|-------|-------------------|
| < 1M | Hash table linear |
| > 1M | Cuckoo filter (switch automático) |

**Recomendações Finais:**
- **Cuckoo Filter**: Usar para >100K alvos (Address) ou >1M alvos (BSGS)
- **Bloom Filter**: Apenas para >10M alvos (caso extremo)
- **Padrão**: Para a maioria dos puzzles (poucos alvos), usar apenas array/hash table simples

**Nota**: O overhead de Cuckoo/Bloom só vale a pena para volumes muito grandes. Para puzzles típicos, não usar filtros é mais rápido.

### 5.3 Kangaroo (Ranges 2^64+)
- Pollard's kangaroo com otimizações
- Distinguished points
- Trap tables sharded (sem lock global)

## 9. CHECKPOINT SYSTEM

### 9.1 Comportamento
- **Auto-save**: A cada 5 minutos (salvamento automático)
- **Emergência**: Ao pressionar Ctrl+C (salvamento forçado)
- **Resume**: Ao iniciar, detecta checkpoint e continua automaticamente

### 9.2 Nomenclatura de Arquivos
```
address_bitXX.ckp   # Address mode (XX = bit range, ex: 21, 32)
bsgs_bitXX.ckp      # BSGS mode (XX = bit range)
kangaroo_bitXX.ckp  # Kangaroo mode (XX = bit range)
```

### 9.3 Formato Único v3
```
[HEADER]
  Magic: 4 bytes "QCHV"
  Version: 4 bytes (uint32)
  Algorithm: "address"|"bsgs"|"kangaroo"

[STATE]
  Range start: 32 bytes
  Range end: 32 bytes
  Current position: 32 bytes
  Threads: 4 bytes
  Timestamp: 8 bytes

[CHECKSUM]
  CRC32 of all above
```

### 9.4 Opções de Checkpoint
```
-c, --checkpoint <arquivo>   Usar arquivo de checkpoint específico
--no-checkpoint              Desabilitar checkpoint
--checkpoint-interval <n>   Intervalo em segundos (padrão: 300)
```

### 9.5 Resume Automático
- Ao iniciar com arquivo de alvo, verificar se existe checkpoint correspondente
- Se existir e válido, perguntar se deseja continuar ou iniciar do zero
- Variável de ambiente QCHVES_RESUME=1 para auto-resume sem perguntar

## 10. AUTO-TUNING

### 10.1 Hardware Detection
```cpp
struct HardwareInfo {
    uint32_t num_cores;
    uint32_t num_physical_cores;
    uint64_t ram_total;
    uint64_t ram_available;
    uint32_t l1_cache;
    uint32_t l2_cache;
    uint32_t l3_cache;
    CPUFeatures features;
    bool is_numa;
};
```

### 10.2 Tunables
- **Batch size**: 256 - 4096 (depende de cache L1)
- **Thread count**: physical_cores ou physical_cores + hyperthreading
- **BSGS table size**: proporcional a RAM disponível
- **Kangaroo fleet size**: proporcional a RAM

### 10.3 Perfis de Auto-Tuning
```
-A, --auto-tune [perfil]    Detectar hardware e ajustar automaticamente

Perfis disponíveis:
  safe      - Usar 50% dos recursos (conservador, menos aquecimento)
  balanced  - Usar 75% dos recursos (recomendado, equilíbrio)
  max       - Usar 100% dos recursos (agressivo, máximo desempenho)

Se não especificado, usa 'balanced' por padrão.
```

### 11.3 Modo Address - Uso
```bash
./address <alvo> [opções]

Argumento:
  <alvo>          Arquivo com endereços ou public keys (ex: Puzzles/71.txt)

Opções:
  -R <modo>      Modo de busca: sequential|backward|both|random
  -l <tipo>      Tipo de busca: compress|uncompress|both
  -t <n>         Número de threads (padrão: auto)
  -A [perfil]    Auto-tuning: safe|balanced|max

Exemplos:
  ./address Puzzles/71.txt -R both -l compress -t 8
  ./address Puzzles/71.txt -R random -A balanced
  ./address Puzzles/25.txt -R sequential -l both -A max
```

### 11.4 Modo BSGS - Uso
```bash
./bsgs <alvo> [opções]

Argumento:
  <alvo>          Arquivo com public keys (ex: Puzzles/71.txt)

Opções:
  -b <n>          Tamanho do range em bits (ex: 32, 40, 48)
  -k <n>          Tamanho da tabela BSGS (padrão: auto por RAM)
  -t <n>         Número de threads (padrão: auto)
  -A [perfil]    Auto-tuning: safe|balanced|max

Tabela -k por RAM:
  4GB  RAM -> -k 512
  8GB  RAM -> -k 1024
  16GB RAM -> -k 2048
  32GB RAM -> -k 4096
  64GB+ RAM -> -k 8192

Exemplos:
  ./bsgs Puzzles/71.txt -b 40 -k 2048 -t 8
  ./bsgs Puzzles/71.txt -b 32 -A balanced
  ./bsgs Puzzles/66.txt -b 66 -k 4096 -A max
```

### 11.5 Modo Kangaroo - Uso
```bash
./kangaroo <alvo> [opções]

Argumento:
  <alvo>          Arquivo com public key (ex: Puzzles/135.txt)
                  ou public key direto em hex

Opções:
  -r <start:end>    Range de busca (ex: 0:FFFFFFFFFFFFFFFF)
  -t <n>            Número de threads (padrão: auto)
  -A [perfil]       Auto-tuning: safe|balanced|max

Exemplos:
  ./kangaroo Puzzles/135.txt -r 0:FFFFFFFFFFFFFFFF -t 8
  ./kangaroo Puzzles/135.txt -r 0:FFFFFFFF -A max
  ./kangaroo 02abcd1234... -r 0:FFFFFFFF -t 4
```

### 11.6 Padrões
- **Output**: FOUND.txt (sempre)
- **Checkpoint**: automatico a cada 5min + Ctrl+C
- **Stats**: a cada 30 segundos (modo automático)
- **Help**: `./address -h` ou `./bsgs -h`

```
  [E] Modo de busca inválido: XX
      Use: sequential, backward, both ou random

  [E] Auto-tune inválido: XX
      Use: safe, balanced ou max
```

### 11.8 Valores Padrão
```
Threads:   auto (detecta CPU)
-t 8      -> usa 8 threads
-k (BSGS): auto (pela RAM)
-A:       balanced (recomendado)
```

### 11.9 Validar Arquivo de Alvo
Ao iniciar, o programa valida o arquivo de alvo:
- Verifica se arquivo existe
- Detecta tipo (endereço BTC / pubkey compress / pubkey uncompress)
- Conta número de alvos no arquivo
- **Valida checksum** de endereços Bitcoin (Base58Check)
- **Ignora linhas inválidas** com warning

**Validação de Endereços BTC:**
```
[E] Linha 5 ignorada: checksum inválido
[E] Linha 10 ignorada: formato inválido (não é base58 nem hex)
[E] Linha 15 ignorada: tamanho inválido (26-35 chars para base58)

[+] Total: 100 alvos válidos, 3 ignorados
```

**Verificação de Public Keys:**
```
[E] Linha 8 ignorada: public key compress inválida (deve ter 66 hex)
[E] Linha 12 ignorada: public key uncompress inválida (deve ter 130 hex)
```
