IMPLEMENTAR - BACKLOG ATUAL DO PROJETO

Objetivo deste arquivo:
- listar o que ainda falta implementar depois da base funcional atual
- separar backlog de performance, algoritmo, infraestrutura e UX
- servir como guia de execução incremental

Estado atual resumido:
- `address`, `bsgs` e `kangaroo` já funcionam
- há backend portátil de referência para secp256k1
- há caminho opcional preparado para `libsecp256k1`
- há checkpoint, resume, save por `Ctrl+C` e paralelização de referência
- a base ainda não representa a implementação final de performance descrita em `migrar.txt`

1. Prioridade Alta - Core criptográfico

- Integrar `libsecp256k1` de fato no ambiente alvo e validar o backend externo em Linux/WSL/Windows.
- Adicionar seleção explícita de backend no runtime ou no build report para facilitar benchmark comparativo.
- Trocar a multiplicação portátil atual por caminho otimizado real no hot path.
- Introduzir batching de derivação/serialização quando possível.
- Reduzir alocações e cópias temporárias em `DerivedKeyInfo`.
- Evitar construção repetida de strings no hot path.
- Separar claramente caminho "benchmark/performance" de caminho "referência/validação".

2. Prioridade Alta - Address mode

- Melhorar paralelização para ranges com largura acima de `u64`.
- Reforçar checkpoint do modo paralelo para registrar posição de forma mais precisa por lane/thread.
- Implementar resume correto também para `both` e `random` no caminho paralelo.
- Adicionar estratégia real de batching por thread, não apenas chunk assignment.
- Medir throughput e ajustar `batch_size` por CPU/cache.
- Adicionar opção de benchmark puro sem escrita de arquivos.
- Revisar progresso/ETA do modo `both` com checkpoint mais fino.

3. Prioridade Alta - BSGS real

- Substituir backend de referência por implementação BSGS de verdade.
- Implementar geração de baby steps.
- Implementar giant steps.
- Implementar tabela real de lookup.
- Adicionar presets de memória/tabela coerentes com `-k`.
- Adicionar estrutura mais eficiente para lookup massivo:
  `hash table` para tamanhos menores
  `cuckoo filter` ou estrutura equivalente para ranges maiores
- Implementar checkpoint específico do estado BSGS.
- Revisar saída de progresso para mostrar construção de tabela e fase de busca.

4. Prioridade Alta - Kangaroo real

- Substituir backend de referência por Pollard Kangaroo real.
- Implementar função de salto.
- Implementar distinguished points.
- Implementar trap tables.
- Implementar checkpoint específico do estado Kangaroo.
- Revisar métricas para mostrar hops, distinguished points e taxa real.
- Definir estratégia de paralelização adequada para herd/fleet.

5. Prioridade Média - Infraestrutura de performance

- Afinidade de CPU por plataforma.
- Detecção melhor de núcleos físicos, caches e NUMA.
- Auto-tuning mais agressivo por perfil.
- Tuning separado para:
  `address`
  `bsgs`
  `kangaroo`
- Benchmarks repetíveis com relatório por backend.
- Modo de benchmark single-thread.
- Modo de benchmark multi-thread.
- Comparação entre backend portátil e backend externo.

6. Prioridade Média - Estruturas de dados e memória

- Pool de memória ou arenas para reduzir overhead.
- Reuso de buffers por thread.
- Evitar `std::string` e `std::vector` em partes muito quentes.
- Introduzir estruturas SoA/AoS onde fizer sentido para SIMD ou batching.
- Reavaliar representação de payloads e serialização para evitar reconstruções.

7. Prioridade Média - CLI e UX

- Mensagens de ajuda mais próximas do plano original.
- Explicar melhor quando o código entra em fallback single-thread.
- Adicionar relatório final do backend usado e do caminho de execução usado.
- Melhorar consistência entre modo exibido e semântica real do backend de referência.
- Adicionar opção clara de benchmark.
- Adicionar opção clara de selecionar backend secp256k1.

8. Prioridade Média - Checkpoint e resume

- Tornar resume mais preciso em modos paralelos.
- Suportar resume forte para `both`.
- Suportar resume forte para `random`.
- Reavaliar formato de checkpoint para estados mais complexos de BSGS/Kangaroo.
- Adicionar testes de corrupção e compatibilidade de checkpoint.

9. Prioridade Média - Testes

- Adicionar testes para:
  `resume` em `backward`
  `resume` em `both`
  `resume` em `random`
  `Ctrl+C` em `bsgs`
  `Ctrl+C` em `kangaroo`
- Adicionar testes de regressão para matcher indexado.
- Adicionar testes com múltiplos alvos.
- Adicionar testes para ranges maiores.
- Adicionar teste opcional do backend `libsecp256k1` quando a lib existir no ambiente.

10. Prioridade Baixa - Organização de código

- Separar a grande implementação de `src/engine/app.cpp` em módulos menores.
- Criar camadas claras:
  `address engine`
  `bsgs engine`
  `kangaroo engine`
  `checkpoint helpers`
  `parallel runtime`
- Isolar melhor estratégia de execução single-thread/paralela.
- Reduzir acoplamento entre CLI, checkpoint e loop de busca.

11. Critérios de pronto para a próxima fase

- Backend `libsecp256k1` validado de fato.
- `address` com throughput claramente melhor que a base atual.
- `BSGS` e `Kangaroo` sem backend de referência residual no fluxo principal.
- Checkpoint/resume confiável em todos os modos.
- Relatório de benchmark reproduzível por ambiente.

12. O que não precisa refazer agora

- Parsing de alvos.
- Formato básico de `FOUND.txt`.
- Build portátil.
- Scripts WSL básicos.
- Checkpoint binário básico com CRC32.

Esses itens já existem e devem ser evoluídos, não reescritos do zero.
